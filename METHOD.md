# שיטת העבודה של הקלסר (קרא לפני שממשיכים)

> מסמך לקלוד ולמשתמשים. מסביר איך הכלי עובד כדי שכל סשן המשך יהיה חלק.
> נעשה עוד החלטות יחד בהמשך — זו הדרך לעשות אותן.

## הרעיון בקצרה
קלסר אינטראקטיבי ב-GitHub Pages (`https://dfarchi.github.io/nipon26/`) שמרכז את כל
תכנון הטיול. כל בחירה היא "צומת" עם אפשרויות, וכל אפשרות נפתחת ל"הסבר" מלא.

## ארכיטקטורת הנתונים — `data.js` הוא הלב
כל ההחלטות, הלילות, המסמכים — ב-`window.TRIP` בתוך `data.js`. הדפים רק מציירים אותו.
לשנות תוכן = לערוך `data.js` בלבד (לא נוגעים ב-HTML).

מבנה אפשרות בהחלטה:
```
{ t:"כותרת", d:"הסבר מורחב",
  w:"איפה (מיקום)", c:"זול|בינוני|יקר",  tr:"השפעת נסיעות (➕/➖/⚪ + טקסט)",
  see:"מה רואים/עושים", tp:"שיקול/טיפ (כולל 💰 רמז תקציבי)",
  l:[{t:"📖/🗺️ תווית", u:"url"}],
  rec: <index של המומלץ או null>,
  decided: <index שנבחר / מערך לרב-בחירה / null> }
```
- `rec` → מציג תג ⭐ מומלץ.
- `decided` → נטען כברירת מחדל במכשיר חדש (ככה שיר רואה את הבחירות הנוכחיות מסומנות).
- `multi:N` → בחירה מרובה (בוחרים N).
- `open:true` → **החלטה פתוחה לדיון עכשיו**. דף ההחלטות מציג כברירת מחדל רק את אלו (כפתור "הצג הכל" חושף את השאר). כשהחלטה נסגרת — מסירים את `open` ממנה והיא נעלמת מהתצוגה הפתוחה (אבל לא נמחקת).
- `noStop:true` → **מאפיין של תחנה, לא תחנה בפני עצמה**. מופיע בדף ההחלטות אבל לא כתחנה נפרדת בציר המסלול (למשל תזמון/מודרך של קומאנו — מתאחדים לתחנה "קומאנו קודו" אחת).
- `cut:true` → **בוטל מהמסלול** (החלטות 2/3/4/5 — היקף הצפון, ניוטו, קקונודאטה, הירוסקי — אחרי שקיצצנו את הצפון ופתחנו בגינזן). לא מוצג בדף ההחלטות ולא בציר המסלול, אבל נשאר ב-`data.js` כתיעוד (לא נמחק — ראו התזכורת למטה).

## מקור אמת לתצוגות
- **דף המסלול + השער** בונים מ-`decided` ב-`data.js` (התוכנית המוסכמת) — לא מ-localStorage.
- **דף ההחלטות** קורא מ-localStorage (סימון אישי במכשיר), עם `decided` כברירת מחדל. לכן ייתכן שהמסלול והדף יראו שונה אם מישהו סימן משהו שעדיין לא הוטמע ב-`decided`.

## כפתור "הסבר ▾"
בכל אפשרות. נפתח ומראה: הסבר, תגיות (📍 איפה · 💰 עלות · 🚆 נסיעות),
👁️ לראות/לעשות, ⚖️ שיקול, ולינקים. לחיצה עליו או על תוכנו לא משנה את הבחירה.

## לינקים
הסביבה של קלוד חסומה מאימות URL (פרוקסי 403), לכן משתמשים בלינקים שתמיד עובדים:
🗺️ Google Maps (תמונות/ביקורות/מיקום) ו-📖 Wikipedia. הפונקציות
`g()`, `m()`, `w()` ב-`data.js` בונות אותם. להימנע מ-URL מספריים שאי-אפשר לאמת.

## לולאת העדכון
1. מקור האמת לתוכן הגדול = מסמכי הדרייב (IDs ב-`TRIP.docs`).
2. יובל/שיר מסמנים ב-`decisions.html` → "העתק בחירות" → שולחים לקלוד בצ'אט.
3. קלוד מעדכן `decided` ב-`data.js` (ואת המסלול/דרייב לפי הצורך), ודוחף.

## דחיפה לגיטהב (חשוב!)
המשתמש רואה רק את GitHub Pages — שום דבר לא נראה לו עד push.
דוחפים דרך GitHub MCP (`push_files`) לענף `main`. **לוודא שכל הקבצים נכללים**
(טעות עבר: שכחתי את `decisions.html` ב-push → 404). Pages בונה מחדש ~1–2 דק';
לבקש מהמשתמש `Ctrl+Shift+R` (קאש).

## יעילות בין סשנים
לקרוא קודם את `MANIFEST.md` + `data.js` — זו המפה. לקפוץ ישירות ל-ID של מסמך
ספציפי בדרייב לפי `TRIP.docs`, בלי חיפוש גורף בכל הדרייב.

## ביטחון
הריפו **פומבי**. אסור להטמיע: PNR/קודי הזמנה, מספרי כרטיסים, כתובות, טלפונים.
רגיש נשאר רק במסמכי הדרייב המוגבלים.

## פתוח עכשיו (להמשך)
אין כרגע החלטות `open:true` — כולן נסגרו. פתוח לתכנון עתידי:
- **בסיס קנסאי**: ~9–10 לילות, נוטה לקיוטו — נסגר בשלב בחירת המלונות.
- **קומאנו**: מתוכנן ספציפית בהמשך (הכיוון: ~4 ימי הליכה ואז אונסן יונומינה).

### מבנה הלילות + קצב האונסן (עדכון 4 ביולי 2026)
עיקרון: **אונסן = פרס/התאוששות אחרי מאמץ**, לא לפניו.
- פתיחה: טוקיו 2 → **גינזן אונסן** (נחיתה אחרי הטיסה). ויתרנו על ניוטו (`cut`).
- דרום טוהוקו: ימהדרה (טיפוס) → **אקיו אונסן**.
- אלפים: ג'יגוקודאי → נקאסנדו (הליכה) → **גרו אונסן** (11ג, נוסף).
- קומאנו: 4 ימי הליכה → **יונומינה**. פוג'י: **קאוואגוצ'יקו**.
- קנסאי: בסיס אחד ~9–10 לילות (נוטה לקיוטו). טוקיו: 2 פתיחה + 4 סיום.
- קומאנו: **עצמאי** (החלטה 14) — חיסכון ~$2,000–4,000 מול מודרך.

## ⏰ תזכורת — מחיקת החלטות סגורות
יובל ביקש: החלטות שכבר הוחלטו **מוסתרות** (לא מציגים `open`), אך כרגע **לא נמחקות**.
בהמשך אפשר למחוק אותן לגמרי מ-`data.js` כדי לנקות — **אבל לבדוק עם יובל לפני כל מחיקה**.
אל תמחק החלטה בלי אישור מפורש.
